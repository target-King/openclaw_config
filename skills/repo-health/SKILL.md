# repo-health

## 目的

检查控制仓库结构是否完整，确保本地仓库和服务器端仓库状态一致，避免同步时才发现缺文件或版本不一致。

## 检查内容

### 仓库结构检查

- 必需目录是否存在
- 每个 agent 是否有 4 个核心文件（AGENTS.md / SOUL.md / USER.md / TOOLS.md）
- 每个 skill 是否有 `SKILL.md`
- `managed-config` 是否齐全
- 记忆脚本是否存在
- `.env.example` 与 `.gitignore` 是否存在

### 同步状态检查

- 本地是否有未提交的修改（`git status`）
- 本地是否有未推送的提交（`git log origin/main..HEAD`）
- 本地分支是否落后于远程（`git fetch && git status`）
- 服务器端仓库是否与远程一致

## 输出

- 通过项
- 缺失项
- 警告项（如未提交修改、未推送提交）
- 建议修复项

## 与标准化同步流程的关系

repo-health 检查应在同步流程的关键节点执行：

```
1. 同步前（本地） — 确认仓库结构完整，无遗漏文件
2. 推送前（本地） — 确认所有修改已提交，无未跟踪的必要文件
3. 拉取后（服务器） — 确认服务器端仓库与远程一致
4. 同步后（服务器） — 确认 .openclaw 目录内容与仓库匹配
```

发现问题时，按 git-sync 技能中定义的标准化流程处理：在本地修复 → 推送 → 服务器拉取 → 重新同步。

## 配套脚本

- `scripts/doctor.ps1` / `scripts/doctor.sh`

## 原则

- 尽量在本地仓库阶段发现问题，不等同步后才报错
- 输出要清楚指出具体路径
- 同步状态不一致时，明确指出差异并给出修复步骤
- 所有修复操作必须遵循标准化同步流程，禁止在服务器端直接修改
